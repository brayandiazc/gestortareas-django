# Convenciones de internacionalización (i18n)

> Cómo manejamos múltiples idiomas y locales en Gestor de Tareas.
> **Última actualización**: 2026-07-02

> **Estado actual: no aplica.** Gestor de Tareas es una aplicación **monolingüe en
> español**; no hay internacionalización implementada. Los textos de la interfaz
> se escriben directamente en español en las plantillas. Esta guía queda como
> referencia reutilizable por si en el futuro se decide soportar varios idiomas
> (Django incluye soporte de i18n con `gettext` y `USE_I18N`).

## Locales soportados

- **Por defecto**: `es` (español), fijo.
- **Adicionales**: ninguno (no aplica actualmente).

## Detección de locale

Orden de prioridad sugerido:

1. Preferencia explícita del usuario (perfil/cookie).
2. Cabecera `Accept-Language`.
3. Locale por defecto.

## Reglas

- Ningún texto visible al usuario se escribe hardcodeado: todo pasa por el sistema de traducción.
- Las claves de traducción se organizan por dominio/feature, no por pantalla.
- Mantener **paridad** entre locales: una clave existe en todos los idiomas soportados.
- Formatos de fecha, número y moneda se localizan según el locale activo.

## Estructura

No existe todavía. Estructura que Django usaría si se activara i18n (archivos
`.po`/`.mo` de `gettext` por idioma):

```text
locale/                     # (propuesto, no existe aún)
├── es/LC_MESSAGES/django.po
└── en/LC_MESSAGES/django.po
```

## Ejemplos

Patrón de Django que se usaría en plantillas y código si se adoptara i18n:

```django
{% load i18n %}
{% trans "Hola" %}
{% blocktrans %}Bienvenido, {{ nombre }}{% endblocktrans %}
```

## Comandos útiles

```bash
# (Propuesto — solo si se activa i18n en Django)
python manage.py makemessages -l es    # extraer cadenas a traducir
python manage.py compilemessages       # compilar .po -> .mo
```
